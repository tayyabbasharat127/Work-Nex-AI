# WorkNex AI — Pre-sale Technical Due Diligence Audit

**Audit date:** 2026-07-12  
**Scope:** `frontend`, `worknex-backend`, `ai-service`, `multi-agent-service`, database schema/migrations, deployment and CI artifacts.  
**Method:** static module-by-module inspection plus repository searches and local test/lint execution. This is not a penetration test and no production infrastructure was examined.

## Executive summary

**Verdict: No — not deployable or sellable as-is. With substantial fixes, yes.** The product has broad functional coverage and a recognizable shared-schema multi-tenant design, but externally exposed AI endpoints trust caller-supplied identity, the biometric webhook can be unauthenticated, tenant isolation is convention-based rather than enforced, and sensitive/runtime artifacts are committed. Frontend lint fails, CI does not run tests, there is no frontend or multi-agent automated test suite, and model/agent evaluation is effectively absent. These are release blockers for a multi-client HR SaaS handling employee data.

MongoDB currently has no role and should not be advertised.

## 6. Multi-agent system

**Architecture map:** a supervisor receives chat, performs deterministic domain routers first, then binds three LangGraph tools: Attendance Agent, Leave Agent, and Performance Agent. Each specialist calls the Express backend using the caller bearer token. The graph loops `supervisor → execute_tools → supervisor` until no tool call, with PostgreSQL checkpoint memory (in-memory fallback enabled). This is hierarchical routing, not broadcast, though three deterministic routers are evaluated for each request.

| Issue | Severity | Location | Fix |
|---|---|---|---|
| No hard graph recursion/tool-call/token/time budget is visible. The conditional graph can cycle as long as the LLM emits tool calls, creating runaway cost/latency risk. | Critical | `multi-agent-service/agents/supervisor.js:161-241` | Set LangGraph recursion limit, max supervisor turns, max specialist calls, deadline/AbortSignal, token/cost budgets, and circuit breakers. |
| Service accepts client identity/token context and exposes conversation thread retrieval; authorization must be airtight, but there is no dedicated service authentication boundary evident at the ingress. | Critical | `multi-agent-service/server.js:88-179` | Verify JWT at ingress, derive user/org/role, authorize thread ownership with org+user composite keys, never trust request context. |
| Memory falls back to in-process storage in production Compose. A PostgreSQL outage silently loses conversational state and changes behavior. | High | `docker-compose.yml:98-100`; `config/memory.js` | Fail closed in production; health/readiness must report unavailable until durable checkpointer works. |
| Tool errors are converted to strings and fed back to the LLM; there is no typed retry classification, redaction, or user-safe failure policy. | High | `supervisor.js:168-225` | Use typed errors, bounded retry only for transient faults, sanitize details, propagate request IDs, and record failure metrics. |
| No automated tests exist for routing, authorization, timeout, memory isolation, loop limits, or agent failure. | Critical | `multi-agent-service` | Add deterministic mocked-LLM tests plus integration/adversarial tests before release. |
| Three specialist agents are mostly thin read-only API wrappers; deterministic routing already handles high-confidence requests. Separate LLM agents add cost and failure modes without clear evidence of improved outcomes. | High | `agents/*`, `routing/*`, `handlers/*` | Use deterministic intent routing + direct typed backend tools for these domains. ## Prioritized action list

### MUST fix before any client/investor technical demonstration

1. Close external AI services: private networking plus verified service/user authentication; remove caller-supplied identity trust and shared privileged aggregate token.
2. Make biometric webhook authentication mandatory with unique device identity, HMAC/replay protection, strict validation, batch limits and rate limiting.
3. Enforce tenant isolation at the database and data-access layers; add composite tenant FKs/RLS and exhaustive two-tenant cross-access tests for every endpoint, job, agent and webhook.
4. Purge committed credentials, production env, logs, uploads and vector runtime data from Git history; rotate every potentially exposed password/key/token.
5. Stop using `Float` for money and repair database constraints/indexes before accepting billing/customer data.
6. Remove HR decision automation claims until models have representative validation, fairness/model-risk review, explainability and mandatory human oversight.
7. Add hard agent limits (turns/tools/tokens/time/cost), authenticated thread ownership, durable-memory fail-closed behavior, and agent tests.
8. Fix frontend lint, token handling and PII console logs; add at least critical-path E2E tests and accessibility checks.
9. Make CI run all test/lint/type/build/migration/security gates on main and PRs. Add release promotion/rollback and backup-restore validation.
10. Establish production observability, incident response, privacy/retention controls, and a deployable production infrastructure definition.

### CAN wait until immediately after security/release blockers

1. Consolidate the duplicate FastAPI chat and Node multi-agent paths; simplify thin specialist agents into deterministic tools where evaluation shows no benefit.
2. Normalize role permission/applicable-role arrays into join tables.
3. Consolidate UI libraries, hooks, repeated dashboard components and legacy TMS code.
4. Improve API naming/envelope consistency and generate documentation from OpenAPI.
5. Move model/vector assets into registries/managed stores and refine bundle/dependency size.
6. Clean encoding, stale comments, screenshots, temporary scripts and documentation duplication.
